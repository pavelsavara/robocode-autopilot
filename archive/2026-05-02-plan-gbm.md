# Architecture: Gradient Boosted Trees (GBM)

*2026-05-02 — Stage 4, ML Architecture 1 of 3*

## Summary

XGBoost / LightGBM on tabular per-tick and per-wave features for three
prediction tasks where the intuition phase already established baselines
with sklearn RandomForest. GBM should strictly dominate RF on the same
feature set because it optimizes a gradient loss directly and handles
feature interactions more efficiently.

## Tasks

### Task A — Fire Power Prediction (regression)

**Target:** `opponent_fire_power` (continuous, [0.1, 3.0])
**Row filter:** `opponent_fired == 1` only (~50–200 rows per battle)
**Metric:** MAE, R²
**Baseline to beat:** RF MAE = 0.21, R² = 0.81 (nb07 honest run)

**Feature exclusions** (from `_loader.FIRE_POWER_LEAKAGE_COLS` + `SCAN_META_COLS`):
- `opponent_fire_power` (target)
- `opponent_bullet_speed` (= 20 − 3·power)
- `mea_for_opponent_bullet` (= asin(8/speed))
- `opponent_inferred_gun_heat` (resets to 1 + power/5)
- All wave-refresh columns: `opponent_wave_*`, `ticks_since_opponent_fired`,
  `escape_angle_coverage`, `gf_current_at_power_*`
- Scan-quality meta-leakage: `scan_coverage_20/50`, `scan_arc_width`,
  `radar_locked`, `radar_turn_direction`, `ticks_between_scans`
  (predict label quality, not opponent behavior — see nb14 Finding 2)

**Additional input features** (added 2026-05-03):
- 20-tick window features: rolling mean/std of all tick features
- 4 strategic axes as context: aggression, range, opponent_family, game_phase

**Expected top features:** `mea_for_our_bullet`, `our_bullet_speed`,
`opponent_energy`, `energy_ratio`, `distance`.

### Task B — Round Outcome Prediction (binary classification)

**Target:** `win_rate >= 0.5` per round (binary)
**Row source:** `scores.csv` — one row per round per perspective
**Metric:** Accuracy, ROC AUC
**Baseline to beat:** RF accuracy = 0.706, majority = 0.506 (nb07)

**Features:** Per-round aggregates from ticks.csv (mean, std of distance,
lat-vel, energy ratio, heading delta, wall distance, fire counts). Plus
per-battle constants from scores.csv (battlefield dimensions, cooling rate).

### Task C — Bot Fingerprinting (multi-class classification)

**Target:** `opponent_bot` — 44 classes (or top-K with min observations)
**Row source:** Per-perspective wave windows (N fires per window, N=5..50)
**Metric:** Top-1 accuracy, weighted F1
**Baseline to beat:** RF 0.349 top-1 at N=20 with 10-stat fingerprint (nb10)

**Features:** 10 wave-window statistics: `mean_power`, `std_power`,
`mean_dist`, `std_dist`, `mean_abs_lat`, `corr_pow_dist`, `power_trend`,
`dist_trend`, `power_autocorr`, `power_range`.

## Architecture

```
XGBoost (or LightGBM) with:
  - n_estimators: 200–1000 (tuned via early stopping)
  - max_depth: 6–10
  - learning_rate: 0.05–0.1
  - subsample: 0.8
  - colsample_bytree: 0.8
  - reg_alpha / reg_lambda: light L1/L2 regularization
```

### Hyperparameter tuning

Use `sklearn.model_selection.RandomizedSearchCV` with 50 iterations,
5-fold GroupKFold on `battle_id` (prevents leaking within-battle structure).

### Why GBM over RF

| Property | Random Forest | XGBoost/LightGBM |
|---|---|---|
| Loss optimization | None (bagged vote) | Gradient descent on loss |
| Regularization | Only max_depth / min_samples | L1, L2, gamma, min_child_weight |
| Missing values | Imputation required | Native handling |
| Speed | Moderate | Faster (histogram-based) |
| Typical lift over RF | — | 5–15% on tabular tasks |

## Training Data

| Task | Source | Rows (estimated at 50×5 sample) |
|---|---|---|
| Fire power | ticks.csv where `opponent_fired==1` | ~15 k |
| Round outcome | scores.csv | ~5 k rounds |
| Fingerprinting | waves.csv windowed | ~10 k windows |

All fit in RAM comfortably. No GPU needed.

## CI Integration

Standard GitHub Actions free runner (Ubuntu, 4 vCPU, 16 GB RAM):
- Training time: < 5 min per task with hyperparameter search
- No GPU required
- Dependencies: `xgboost`, `lightgbm`, `scikit-learn`, `pandas`, `numpy`

GPU runners require GitHub Team/Enterprise plan ($0.07/min) — not needed
for GBM. The free tier's 4 vCPU is sufficient since both XGBoost and
LightGBM parallelize tree building across cores.

## Output Artifacts

Per task:
- `models/<task>/model.json` — serialized XGBoost model (JSON format)
- `models/<task>/metrics.json` — train/test metrics, feature importance
- `models/<task>/feature_importance.csv` — per-feature gain/weight/cover
