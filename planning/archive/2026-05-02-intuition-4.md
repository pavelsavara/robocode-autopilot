## What the new data confirmed

### 1. Wave-fingerprint findings replicate exactly
Notebook 08 was already run on a single season in Phase 3. With both seasons
loaded it returns **identical** numbers to the rounding shown in Phase 3:

- 737,381 wave events, 50 families — same.
- Pipeline physics residuals: `speed=0.015`, `mea=5×10⁻⁵`, `flight=0.007` — same.
- Pooled power-distance correlation `r = -0.337` — same.
- Most aggressive: Saguaro (1.99 ± 0.80), Pris (1.84), Ascendant (1.81),
  Seraphim (1.81) — same ordering.
- Most adaptive: Garm (-0.79), Domogled (-0.48), RougeDC (-0.43), Foilist
  (-0.40), Neuromancer (-0.37) — same ordering, same magnitudes.
- PCA: PC1=46%, PC2=19%, total=66% — same.

This is the strongest evidence yet that wave-fingerprints are a stable bot
characterisation. Doubling the underlying recording pool changes neither the
ranks nor the magnitudes.

### 2. Fire-power prediction is degenerate
**Notebook 07 (cell 9): RandomForest predicts opponent fire power with R² = 1.000,
MAE = 0.000 on 11k held-out samples.** Top features:
`mea_for_opponent_bullet` (0.34), `opponent_bullet_speed` (0.29),
`opponent_inferred_gun_heat` (0.29).

This is **not learning** — it's three identities. Knowing
`opponent_bullet_speed` *exactly* determines `power = (20 − speed)/3`, and
`mea = asin(8 / speed)` is also a direct function of speed. The model is doing
algebra, not behavioural modelling.

**Action:** the "fire-power prediction" task as currently set up is meaningless.
For a real prediction task, the wave row already has `wave_bullet_power` —
the meaningful question is "given opponent state at tick *T* (before the fire
event at tick *T+k*), can we predict the power of the eventual fire?" That
needs a join from wave events back to ticks at *T-1, T-2, …* with the speed/MEA
columns dropped, not the same-tick state.

### 3. Fire-event prediction is ALSO degenerate (new finding)
**Notebook 04 (cell 4): all four classifiers — DecisionTree(d=5), DecisionTree
(unlimited), RandomForest, LogisticRegression — score 1.0000 train AND test
on the `opponent_fired` task.** Top feature: `opponent_wave_distance` (0.37).

This is leakage from a different angle. `opponent_wave_distance` is the
distance the latest detected bullet has travelled — it resets to 0 the moment
`opponent_fired = 1`. So the "predictor" is literally "did wave_distance just
reset?", which is identical to `opponent_fired`. Phase 1's much smaller
dataset reported 98.88% accuracy on the same task, headlined as "fire timing
is trivial — gun heat dominates"; the truth is that gun heat **is** a real
predictor (we saw 88% feature importance there) but
`opponent_wave_distance` / `escape_angle_coverage` / `ticks_since_opponent_fired`
all carry direct leakage from the wave-tracking features. The model is
fingerprinting the wave-reset event, not predicting it.

**Action:** rebuild the fire-prediction task with wave-derived columns
explicitly excluded — only `opponent_inferred_gun_heat`, kinematics, and
geometry. The 88.45% gun-heat-dominated baseline from Phase 1 is the right
reference number.

### 4. Net-damage classification on scores.csv is a real signal
**Notebook 04 (cell 12): RandomForest on per-round score features predicts
"did this round end positive net damage" with 0.830 accuracy** versus a 0.524
majority baseline. 5,250 rounds. This is honest — the inputs are
round-aggregated outcomes (damage_dealt, damage_received, hit_rates) so the
task is essentially "given the round's damage pattern, classify the sign of
net damage". The 0.83 ceiling tells us mid-round outcomes are noisy enough to
not perfectly predict the sign — useful as a sanity baseline for any future
"can we predict outcome from inputs alone" model on the much larger 136k
score row pool.

### 5. Movement clustering finds 4 archetypes, but the boundary is fuzzy
**Notebook 05 (cell 9): silhouette peaks at K=4 with score 0.202**, then drops
to 0.170 at K=5. Cluster membership:

- **Cluster 0** (4 bots): BeepBoop, DrussGT, Roborio, Seraphim — top-tier
  movers, the "wavesurfer" archetype.
- **Cluster 1** (6 bots): Holden, Neuromancer, Pris, Saguaro, XanderCat,
  YersiniaPestis — high lat-vel bots with shorter direction-change times.
- **Cluster 2** (25 bots): the "generic competent" majority including Diamond,
  Shadow, Cassius, Dookious, Gilgalad — most of the established mid-pack.
- **Cluster 3** (14 bots): Ali, Cyanide, Firestarter, Foilist, Phoenix,
  PowerHouse, Tomcat, X2 — fast, decisive movers (high vel-mean, short
  direction-change times).

Silhouette of 0.20 is "structure exists but boundaries are soft" — these are
real groupings, not a clean partition. The two top-tier wavesurfers landing
in the same cluster (BeepBoop & DrussGT) is a noteworthy validation.

### 6. Per-tick movement autocorrelation is essentially zero
**Notebook 05 (cell 13):** lateral-velocity autocorrelation peaks at lag 17–53
ticks with **|r| ≤ 0.03 for every bot**, both periodic and "random". The
takeaway: at tick granularity, no bot is meaningfully periodic — the lat-vel
trace is dominated by noise relative to lag-correlated structure. Movement
prediction at horizons ≥5 ticks must rely on **state context** (distance,
energy, relative heading, gun-cooldown) rather than naive autoregression.

### 7. Bot fingerprinting from movement alone is feasible but expensive
**Notebook 06 (cell 6):** 50-tick window, 46-class classification → **45.0%
top-1 / 71.9% top-5** accuracy (random baseline 2.2%). Window study (cell 9):

| Window | Top-1 accuracy |
|---:|---:|
| 10 ticks | 0.32 |
| 20 ticks | 0.37 |
| 30 ticks | 0.39 |
| 50 ticks | 0.43 |
| 100 ticks | 0.54 |

Two things stand out:
- **The easy bots are very easy.** Neuromancer 0.99, BeepBoop 0.93, DrussGT 0.85,
  Tomcat 0.84 — these have such distinctive movement that 50 ticks is enough.
- **The hard bots are essentially invisible** — Seraphim, Wintermute, BlackBox,
  Midboss, Nene, PowerHouse all score 0.00 in this experiment. Most of those
  appear in Cluster 2 (the "generic competent" group from §5), confirming
  that cluster IS a real "movement-indistinguishable" set.

Compared to wave fingerprints (§1, 50 fires ≈ 10s of game time → 66%
PC1+PC2 separation), movement fingerprints need 100 ticks (~10s of game time
also) to reach 54% top-1. **Wave fingerprints are the more efficient identity
signal**, and they only need the opponent to fire — which they will,
constantly.

### 8. Adaptation across rounds: DrussGT confirmed top
**Notebook 05 (cell 20):** KS-test for "lat-vel distribution in rounds 0-4 vs
rounds 30-34" finds:

| Bot | KS-stat | p-value |
|---|---:|---:|
| **DrussGT 3.1.7** | **0.172** | 4×10⁻⁵⁵ |
| XanderCat 12.9 | 0.084 | 7×10⁻⁵ |
| Foilist 1.3.1 | 0.067 | 9×10⁻⁵ |
| Shadow 3.83c | 0.065 | 7×10⁻⁴ |
| Roborio 1.2.4 | 0.064 | 3×10⁻²⁰ |

DrussGT's distribution shift is **2× larger** than any other bot. This is
the famous DrussGT "movement learner" in action — its movement at round 30
is statistically a different distribution from round 0 because it has
adapted to its opponent. Most bots (Diamond, Phoenix, Holden) score KS
≈ 0.02 with non-significant p — their movement is round-invariant.

This is a high-value labelling signal: any future "is the opponent adapting"
detector can be calibrated against this exact KS measurement.

### 9. Cross-feature redundancy, now confirmed at scale
**Notebook 02 (cell 5):** the top-correlated feature pairs at 1M ticks are:

| Pair | r |
|---|---:|
| `opponent_guess_factor` ↔ `our_lateral_velocity` | **1.000** |
| `distance` ↔ `distance_norm` | 1.000 |
| `gf_current_at_power_1_5` ↔ `gf_current_at_power_2` | 0.999 |
| `gf_current_at_power_1` ↔ `gf_current_at_power_1_5` | 0.999 |
| `opponent_wave_distance` ↔ `ticks_since_opponent_fired` | 0.998 |
| `opponent_wave_distance` ↔ `opponent_wave_remaining` | -0.998 |

`opponent_guess_factor ≡ our_lateral_velocity` (r=1.000) is by **definition**:
GF is computed from lat-vel and the bullet's max-escape-angle. Worth noting:
when training movement-prediction models, drop one of the two — they carry
zero independent information.

The three GF columns at different power levels are also nearly perfectly
correlated (r > 0.999) — keep `gf_current_at_power_2` and drop the other two.

The `opponent_wave_distance` triplet (with `ticks_since_opponent_fired` and
`wave_remaining`) is the leakage source identified in §3.

## Pipeline / infrastructure changes from this run

1. **Stage 2 silent-failure bug fixed** in `Main.java` and
   `process-recordings.yml`: `pipeline` exits 2 if `processed == 0`, and
   `actions/upload-artifact` now uses `if-no-files-found: error`. A future
   corrupt `.br` chunk will fail loudly instead of producing a green run with
   empty CSVs.
2. **`_loader.py` made multi-root and path-cached.** `build_robot_index()`
   now accepts a single root or an iterable of roots, walks all of them with
   `rglob('ticks.csv')`, and stores the discovered directory per
   `(battle, robot)` in a module-level cache. `load_stratified()` reads from
   that cache first. This is what made re-running the notebooks against
   chunked layouts (`output/csv-25250609244/csv-chunk-N/<battle>/<robot>/`)
   trivial — no notebook code changes needed.
3. **Directory junctions over chunked downloads.** A small one-shot PowerShell
   command junctions every `<battle_id>` directory from both seasons'
   `csv-chunk-N/` subtrees into `output/csv/`. 1944 junctions, zero
   collisions across the two seasons. Notebooks see a flat tree.
4. **`download-csv-run.ps1` rewritten with retry + stall-tolerant timeout.**
   Original script had no retry; `gh run download` writes 0 bytes for ~60s
   per chunk before flushing the unzip in one burst, which fooled my first
   attempt at stall detection. The fixed version uses a 15-minute hard
   timeout per attempt and 8 retries with 5s backoff.

## Concrete recommendations for next phase

1. **Rebuild fire-prediction task** (notebook 04) with all wave-derived
   columns excluded: `opponent_wave_distance`, `opponent_wave_remaining`,
   `opponent_wave_eta`, `ticks_since_opponent_fired`, `escape_angle_coverage`,
   `our_wave_*`. The honest predictor is `opponent_inferred_gun_heat`. Aim
   for the 88-95% accuracy band, not 100%.
2. **Rebuild fire-power task** (notebook 07) with `opponent_bullet_speed`,
   `mea_for_opponent_bullet`, and `opponent_fire_power` excluded — those are
   algebraic identities of the answer. Use opponent state at *T-k* to predict
   the power of the next fire event at *T*.
3. **Drop redundant features in any model** that uses `our_lateral_velocity`:
   `opponent_guess_factor` is identical. Keep one of `distance` /
   `distance_norm`. Keep one GF column out of the three power levels.
4. **Port notebooks 05 and 07 to the current schema.** They're the highest-
   value rewrites — 05 owns position-prediction and adaptation, 07 owns
   round-outcome modelling, and both currently produce only partial output.
5. **Use wave fingerprints, not movement, for online opponent ID.** §7 shows
   movement needs 100 ticks for 54% top-1 across 46 classes; §1 shows wave
   fingerprints achieve 66% PC1+PC2 spread on six stats. After ~50 fires
   (≈10s of fire activity), bot identity is essentially solved.
6. **DrussGT's KS=0.17 is a labelling target.** Any "is the opponent learning"
   detector should be calibrated to fire when the running KS distance from
   round 0's lat-vel distribution exceeds ~0.10. That captures DrussGT,
   XanderCat, Foilist, Shadow — the known adaptive bots — without false-
   firing on the static-movement majority.
